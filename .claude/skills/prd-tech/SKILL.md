# PRD Tech — Technical Design & Implementation Plan

Transform a business PRD (`prd-product-*.md`) into a technical specification with architecture decisions and an ordered implementation task breakdown. The goal: produce a technical plan so clear that a developer or AI agent can implement each task without architectural guesswork.

**Core philosophy:** Every technical decision must trace back to a business requirement from the product PRD. No gold-plating, no speculative abstractions. Design exactly what the business needs — nothing more.

**Input:** A `tasks/prd-product-{feature}.md` file (generated by `/prd-product`)
**Output:** A `tasks/prd-tech-{feature}.md` file ready for `/ralph` conversion

---

Before engaging with the user, build a complete technical picture.

- Relevant existing code found (models, services, routes)
- Canonical entities involved (from the terminology table)
- Potential technical constraints or conflicts detected
- Initial architectural direction you're considering

## Phase 1: Technical Interview

A focused interview to resolve technical decisions that can't be derived from the business PRD alone. This should be shorter than the product interview — many answers are already in the business PRD or can be inferred from project conventions.

### Rules

**Ask only what you can't infer.** If the project conventions (CLAUDE.md, existing code) make a decision obvious, don't ask — state your assumption and move on. Only ask when there's genuine ambiguity.

**Group questions by decision area.** Don't mix data model questions with UI questions. Each round should focus on one technical domain.

**Propose, don't just ask.** For each question, offer your recommended approach with reasoning. The user can accept quickly or redirect: "I'd suggest X because Y. Does that work, or do you have a different preference?"

### Interview Areas

**1. Data Model & Schema**
- What new entities/tables are needed?
- How do they relate to existing entities (especially canonical ones defined in CLAUDE.md)?
- What are the cardinalities? (1:1, 1:N, N:M)
- Are there enums or status fields? What are their values and transitions?
- Any computed/derived data? Where does it live?
- Does any data need soft deletes?

**2. Service Layer & Business Logic**
- Which service classes are needed?
- Where does complex logic live? (service vs model vs job)
- Are there async operations? (queued jobs, events)
- What external integrations are involved? (check CLAUDE.md for configured integrations)
- Caching strategy for frequently accessed data?

**3. API & Routes**
- What routes/endpoints are needed?
- Filament admin resources? Which panels?
- Inertia pages for the frontend?
- API resources for external consumption?
- Authorization: what policies are needed?

**4. Search & Indexing**
- Does this feature need search? (check CLAUDE.md for search stack)
- What fields should be searchable? Filterable? Sortable?
- Any faceted filtering requirements?

**5. Edge Cases & Error Handling**
- What failure modes exist at the technical level?
- Concurrency issues? (race conditions, optimistic locking)
- Data integrity constraints beyond what the DB enforces?
- What happens during partial failures in multi-step operations?

### Confidence Tracking

Read `.claude/skills/references/interview-methodology.md` → **Confidence Tracking** for the mechanism.

**This skill's threshold: 90%** across all relevant technical areas. Technical PRDs can tolerate slightly more ambiguity than business PRDs because implementation itself will resolve some details — but architectural decisions must be firm.

## Phase 2: Technical PRD Generation

### Structure

#### 1. Overview
- Reference to the source business PRD (`tasks/prd-product-{feature}.md`)
- One-paragraph technical summary of what's being built
- Key architectural decisions made during the interview

#### 2. Data Model

For each new entity:

```markdown
### [EntityName]

**Table:** `table_name`
**Model:** `App\Models\EntityName`

| Column | Type | Nullable | Default | Notes |
|--------|------|----------|---------|-------|
| id | ulid | no | auto | PK |
| name | string | no | — | — |
| status | enum(StatusEnum) | no | Draft | Cast to PHP enum |
| ... | ... | ... | ... | ... |

**Relationships:**
- belongsTo: ParentModel (via `parent_id`)
- hasMany: ChildModel

**Indexes:**
- unique: [col_a, col_b]
- index: [status]

**Enums:**
- StatusEnum: Draft, Active, Archived
```

Include an entity relationship summary (text-based, not diagrams) showing how new entities connect to existing canonical entities.

#### 3. Service Layer

For each service:
- Class name and namespace
- Public methods with signatures and brief description
- Dependencies (injected services, repositories)
- Business rules implemented (reference US-XXX from the product PRD)

#### 4. API & Routes

For each route group:
- HTTP method + path
- Controller + method
- Request validation (Form Request class)
- Response format (Resource class or Inertia page)
- Authorization (Policy method)
- Which business user story it serves (US-XXX)

#### 5. Admin UI Resources (if applicable)

> **Include this section ONLY if the project uses an admin panel framework (e.g. Filament, Nova, Backpack).** Skip entirely if the project has no admin panel.

For each admin resource:
- Resource class and panel
- Table columns and filters
- Form fields and validation
- Actions (custom actions if any)
- Relation managers

#### 6. Search Configuration (if applicable)

> **Include this section ONLY if the feature requires search AND the project has a search engine configured.** Skip entirely if search is not needed.

- Models with search indexing
- Searchable fields definition
- Index settings (filterable, sortable attributes)
- Faceted filtering configuration

#### 7. Implementation Tasks

This is the ordered, dependency-aware breakdown of work. Each task is an atomic unit that can be implemented and verified independently.

**Format:**

```markdown
### T-001: [Task title]
**Type:** migration | model | service | controller | filament | frontend | test
**Depends on:** — (or T-XXX)
**Implements:** US-001, US-002 (reference to business PRD user stories)

**Description:**
What to build, specifically. Include the key details a developer needs.

**Acceptance Criteria:**
- [ ] Specific verifiable criterion
- [ ] Another criterion
- [ ] Typecheck passes
- [ ] Tests pass
```

**Ordering rules:**
1. Migrations and enums first (schema foundation)
2. Models and relationships second
3. Services and business logic third
4. Controllers, routes, and policies fourth
5. Filament resources / UI fifth
6. Integration and end-to-end tests last

**Sizing rules:**
- Each task must be completable in one focused session (one Ralph iteration)
- If a task needs more than 2-3 sentences to describe, it's too big — split it
- Each task must be independently verifiable

**Mandatory acceptance criteria:** Read `.claude/skills/references/acceptance-criteria.md` → **Mandatory Criteria by Type**.

#### 8. Technical Risks & Open Questions
- Known risks with mitigation strategies
- Performance concerns and benchmarks to hit
- Genuine unknowns that will be resolved during implementation

### Traceability Matrix

After the main sections, include a traceability matrix:

```markdown
## Traceability: Business US → Implementation Tasks

| Business US | Description | Implementation Tasks |
|-------------|-------------|---------------------|
| US-001 | [title] | T-001, T-003, T-007 |
| US-002 | [title] | T-002, T-004 |
```

Every business US must be covered by at least one implementation task. If a business US has no corresponding task, something is missing.

### Incremental Writing Protocol

Read `.claude/skills/references/incremental-writing-protocol.md` and follow the protocol.

**Skill-specific details:**
- **Document title prefix:** `# PRD Tech: [Feature Name]` with `> Source: tasks/prd-product-{feature}.md`
- **Sections:** 8+ total (optional sections: Admin UI, Search Configuration)
- **Sections that may need batching:** Implementation Tasks (Section 7) — max 10 tasks per `Edit` (tasks are more detailed, so smaller batches)
- **Output path:** `tasks/prd-tech-{feature}.md`

Progress feedback uses `8+` as denominator to account for optional sections.

## Phase 3: Review & Validation

After generating the technical PRD:

1. **Verify traceability** — Every business user story (US-XXX) from the product PRD must map to at least one implementation task (T-XXX). Flag any gaps.
2. **Verify dependency order** — No task should depend on a task with a higher number. Walk through the order and confirm it makes sense.
3. **Highlight key architectural decisions** — "I chose X over Y because Z. Does this align with your thinking?"
4. **Flag risk areas** — "The trickiest part is X — here's how I've structured the tasks to minimize risk."
5. **Invite feedback** — "Anything that feels wrong architecturally or that I've oversized/undersized?"

Iterate until the user confirms.

**After confirmation, suggest:** "The technical PRD is ready. When you want to generate the Ralph execution file, run `/ralph`."

---

## Output

- **Format:** Markdown (`.md`)
- **Location:** `tasks/`
- **Filename:** `prd-tech-{feature}.md` (kebab-case, matching the product PRD name)
- **How it's written:** Always use the **Incremental Writing Protocol**. Never produce the full PRD in a single `Write` call.

**Important:** Do NOT start implementing. The output is the technical PRD only. The next step is `/ralph` for autonomous execution.
